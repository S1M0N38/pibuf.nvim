INTRODUCTION                                                             *pibuf*

type definitions, and automated workflows.

FEATURES                                                        *pibuf-features*

pibuf.nvim provides a template with the following features:

Plugin Structure: ~

  A well-organized plugin structure is essential for maintainability,
  readability, and following Neovim community conventions. This template
  provides a proven directory layout that separates concerns and makes your
  plugin easy to navigate for both users and contributors.

  - Organized file layout with lua/pibuf/init.lua for core logic
  - plugin/pibuf.lua for plugin manager integration
  - lua/pibuf/health.lua for health check functionality
  - lua/pibuf/types.lua for LuaCATS type definitions
  - Standard directory structure following Neovim conventions

Development Environment: ~

  Modern plugin development requires sophisticated tooling to catch errors
  early, provide intelligent code completion, and maintain code quality. The
  integrated development environment setup ensures you have the same level of
  support that users expect from professional development tools.

  - Lua Language Server integration with autocompletion and diagnostics
  - LuaCATS annotations support for type checking and documentation
  - lazydev.nvim integration for enhanced plugin development experience
  - lazy.nvim configuration examples for local development workflow

Testing Framework: ~

  Reliable plugins require comprehensive testing to prevent regressions and
  ensure consistent behavior across different Neovim versions and
  configurations. The testing framework provides the foundation for building
  confidence in your plugin's stability and correctness.

  - mini.test framework with child-process isolation
  - Test files in tests/ directory with automated discovery
  - lazy.minit test runner (tests/minit.lua)

Documentation System: ~

  Quality documentation is crucial for plugin adoption and user experience.
  Integrating with Neovim's built-in help system ensures users can access
  documentation offline and in a familiar format, following established
  conventions that Neovim users expect.

  - Vim help documentation using vimdoc format in doc/ directory
  - Integration with Neovim's built-in help system
  - Documentation accessible via :help commands within editor

Automation and Workflows: ~

  Automated workflows reduce manual overhead, prevent human errors, and ensure
  consistent quality across releases. Continuous integration catches issues
  before they reach users, while automated publishing streamlines the release
  process and maintains professional standards.

  - GitHub Actions workflows for continuous integration
  - Type checking with Lua Language Server validation
  - Automated releases based on conventional commit specifications
  - LuaRocks and GitHub package publishing with semantic versioning
  - Minimal reproduction environment for debugging with repro/repro.lua

REQUIREMENTS                                                *pibuf-requirements*

Before using pibuf.nvim as a template, ensure you have the following:

- Neovim ≥ 0.12.2
- StyLua (https://github.com/JohnnyMorganz/StyLua) for code formatting
  and linting
- LuaLS (https://github.com/LuaLS/lua-language-server) for type checking
  via LuaCATS annotations
- git (https://git-scm.com/) for version control and lazy.nvim bootstrap
- Make (https://www.gnu.org/software/make/) for build and test commands

Optional: ~

- lazydev.nvim (https://github.com/folke/lazydev.nvim) for Lua LSP
  configuration during plugin development

INSTALLATION                                                *pibuf-installation*

pibuf.nvim is primarily designed as a template for creating new plugins.

Using as a template: ~

1. Click "Use this template" on GitHub to create a new repository
2. Clone your new repository and customize for your plugin:
   - Replace "pibuf" with your plugin name throughout the codebase
   - Replace "S1M0N38" with your GitHub username
   - Update plugin description and functionality
Development installation: ~

For development and testing, install using lazy.nvim:

>lua
  {
    {
      "your-plugin.nvim",
      dir = "/path/to/your-plugin.nvim",
      branch = "main",
      lazy = false,
      opts = {},
      keys = {
        {
          "<leader>rb",
          "<cmd>Lazy reload your-plugin.nvim<cr>",
          desc = "Reload your-plugin.nvim",
          mode = { "n", "v" },
        },
      },
    },

    {
      "folke/lazydev.nvim",
      ft = "lua",
      opts = {
        library = {
          "pibuf.nvim",
        }
      },
    },

  }
<

CONFIGURATION                                                     *pibuf-config*

pibuf.setup({opts}) ~

The `pibuf.setup()` function configures the plugin with user-provided options.
Available options:

>lua
  {
    name = "John Doe", -- The name of the user (string, default: "John Doe")
  }
<

The configuration system demonstrates proper option handling with defaults
and user customization. When building your plugin, extend this pattern to
support your specific configuration needs.

Example configuration: ~
>lua
  require("pibuf").setup({
    name = "Jane Developer",
  })
<

COMMANDS                                                        *pibuf-commands*

pibuf.nvim provides the following commands for demonstration and testing:

:Pibuf [subcommand] ~
  Execute a pibuf.nvim subcommand. Available subcommands:
  - `hello`: Display greeting message to the user
  - `bye`: Display farewell message to the user

  If no subcommand is provided or an invalid subcommand is used, an error
  message will be displayed with usage information.

:checkhealth pibuf ~
  Run health checks to verify pibuf.nvim setup and configuration.
  This command validates:
  - Plugin installation and loading
  - Configuration option validation
  - Core functionality testing
  - Development dependencies availability

API                                                                  *pibuf-api*

pibuf.nvim exposes a minimal API focused on setup and core functionality:

                                                                 *pibuf.setup()*
pibuf.setup({opts}) ~
  Set up the plugin with the given options. See |pibuf-config| for
  available options. This is the primary public function exposed by the
  main module.

                                                                 *pibuf.hello()*
pibuf.hello() ~
  Say hello to the user using the configured name. Returns a string with
  the greeting message and notifies it via `vim.notify()`.

  Return format: ~
  "Hello [configured name]"

                                                                   *pibuf.bye()*
pibuf.bye() ~
  Say bye to the user using the configured name. Returns a string with
  the farewell message and notifies it via `vim.notify()`.

  Return format: ~
  "Bye [configured name]"

Usage example: ~
>lua
    -- Basic usage
    require("pibuf").setup({ name = "Developer" })

    -- Call functions directly
    local greeting = require("pibuf").hello()
    local farewell = require("pibuf").bye()

    -- Access configuration
    local Config = require("pibuf.config")
    print("Current name: " .. Config.name)
<

DEVELOPMENT                                                  *pibuf-development*

pibuf.nvim includes a development workflow for plugin creation:

Testing: ~

Run tests using mini.test via lazy.minit:
>bash
  # Run all tests
  make test

  # Run a single test file
  make test-one MODULE=pibuf

  # Run explicit file
  nvim -l tests/minit.lua --minitest tests/pibuf_spec.lua
<

Health Checks: ~

Validate plugin setup and functionality:
>vim
  :checkhealth pibuf
<

Minimal Reproduction: ~

Debug issues using the included reproduction environment:
>bash
  # Start Neovim with minimal config
  nvim -u repro/repro.lua
<

Code Formatting: ~

The template uses StyLua for consistent Lua code formatting:
>bash
  # Check formatting
  make lint

  # Auto-format all Lua files
  make format
<

Formatting settings are defined in `stylua.toml` (2-space indentation,
double quotes, 120 column width, Unix line endings).

Generate Help Tags: ~

After modifying `doc/pibuf.txt`, regenerate the tags file:
>vim
  :helptags doc/
<

Or from the command line:
>bash
  nvim -c "helptags doc/" -c "q"
<

Release Automation: ~

The template includes automated workflows for:
- Running tests on every push (GitHub Actions)
- Type checking with LuaLS
- Automated releases based on conventional commits
- LuaRocks package publishing via release-please

Use conventional commit format for automatic versioning:
- `feat:` for new features (minor version bump)
- `fix:` for bug fixes (patch version bump)
- `BREAKING CHANGE:` for breaking changes (major version bump)

A more detailed explanation for conventional commits can be found at
https://www.conventionalcommits.org/.

CI/CD TOKENS                                                    *pibuf-tokens*

The CI/CD workflows require the following secrets to be configured in your
GitHub repository. Navigate to Settings > Secrets and variables > Actions and
add each secret as a repository secret.

GITHUB_TOKEN: ~

Automatically provided by GitHub Actions. No manual setup required.
Used by the CI workflow for StyLua formatting checks.

PAT (Personal Access Token): ~

A Fine-grained Personal Access Token used by release-please to create
release pull requests and GitHub releases. Using a PAT instead of the
default `GITHUB_TOKEN` ensures that release PRs trigger other workflow
runs (e.g., CI checks on the PR).

To create a Fine-grained PAT:

1. Go to https://github.com/settings/personal-access-tokens/new
2. Set a descriptive name (e.g., "release-please")
3. Set expiration as needed
4. Under "Repository access", select "Only select repositories"
   and choose your plugin repository
5. Under "Permissions" > "Repository permissions", grant:
   - Contents: Read and write
   - Pull requests: Read and write
6. Generate the token and copy the value
7. Add it as the `PAT` secret in your repository settings

You may also need to enable "Allow GitHub Actions to create and approve
pull requests" under Settings > Actions > General.

LUAROCKS_API_KEY: ~

An API key from luarocks.org used by luarocks-tag-release to automatically
publish your plugin to LuaRocks when a release is created.

To obtain a LuaRocks API key:

1. Create an account at https://luarocks.org
2. Go to Settings > API Keys
   (https://luarocks.org/settings/api-keys)
3. Generate a new API key
4. Add it as the `LUAROCKS_API_KEY` secret in your repository settings

Summary: ~

>text
  Secret name        Used by                Setup required
  ----------------------------------------------------------------------
  GITHUB_TOKEN       StyLua (ci.yml)        Automatic
  PAT                release-please         Manual (Fine-grained PAT)
  LUAROCKS_API_KEY   luarocks-tag-release   Manual (luarocks.org)
<

TROUBLESHOOTING                                          *pibuf-troubleshooting*

First, run the health check to identify issues:
>vim
  :checkhealth pibuf
<

If you encounter problems, follow these steps to reproduce the issue
in a clean environment:

1. Clone pibuf.nvim and cd into the directory
2. Run `nvim -u repro/repro.lua`
3. Run `:checkhealth pibuf`
4. Try to reproduce the issue
5. Report the repro.lua and any error messages

If you need to programmatically run lua code for testing, you can use the
repro/repro.lua file for configuration and use the `-l` flag to load the
script file, i.e.:

>bash
  nvim -u repro/repro.lua -l <script.lua>
<

or directly run the command with the `-c` flag:
>bash
  # Run the command after loading the config in repro/repro.lua file
  nvim -u repro/repro.lua -c "Pibuf hello"

  # Run in headless mode and quit after running the command
  nvim --headless -u repro/repro.lua -c "Pibuf hello" -c "q"
<


Manual Testing: ~
